# RAG 연동 개요

이 문서는 `rag/`를 Backend, AI, Infra 팀에 설명할 때 먼저 보여주는 한 장 요약이다.
현재 구현과 이후 연동 기준을 한 번에 정리한다.

## 1. RAG의 현재 책임

현재 `rag/`는 아래 영역을 담당한다.

```text
문서 입력
  -> 텍스트 추출
  -> chunking
  -> embedding
  -> Chroma 저장

이메일 입력
  -> template candidate 검색
  -> 추천 결과 저장
```

즉 역할을 나누면:

```text
Retrieval 역할
- knowledge ingest / search
- template index / match

Generation 역할
- draft
```

현재는 한 FastAPI 앱 안에 같이 들어 있다.

## 2. Backend가 최종적으로 알아야 하는 외부 API

현재 외부 연동 기준은 아래 4개다.

```text
1. POST /knowledge/ingest
2. POST /templates/index
3. POST /templates/match
4. q.2rag.draft
```

예전처럼 중간 결과를 주고받는 API는 외부 계약에서 숨기는 방향이다.

```text
외부에 숨기는 내부 단계
- /extract-text
- /canonical/email
- /canonical/template
- /knowledge/index
```

즉 Backend는 `중간 산출물`을 몰라도 되게 정리한다.

## 3. 사용자 관점 흐름

### 3-1. 온보딩 / 템플릿 생성

```text
사용자 입력
- 회사소개
- 업종
- 메일 어조
- FAQ
- 업로드 문서
- 카테고리

Backend
-> 원본 저장
-> /knowledge/ingest
-> q.2rag.draft
-> 템플릿 DB 저장
-> /templates/index
```

주의:

현재 `draft`는 일반 답장 초안 생성이 아니라 온보딩 템플릿 생성 계약을 의미한다.

여기서 문맥은 두 종류로 나뉜다.

```text
고정 문맥
- company_description
- domain
- industry_type
- mail_tone

검색 문맥
- FAQ
- 업로드 문서(manual)
- RAG는 `category_name + industry_type + company_description`를 query로 knowledge search를 수행하고,
  검색된 chunk를 고정 `rag_context`와 합쳐 템플릿 생성 문맥으로 사용한다.
```

### 3-2. 이메일 수신 / 템플릿 추천

```text
메일 수신
-> Backend가 메일 저장
-> AI가 summary / intent / domain 생성
-> templates.match 수행
-> 추천 결과 저장
```

여기서 `templates.match`는 사용자가 메일을 열 때마다 즉시 계산하는 방식보다,
메일 수신 파이프라인에서 미리 수행하는 방식이 더 자연스럽다.

## 4. 현재 코드 기준 실제 상태

현재 `rag/`는 단독으로는 동작 확인 가능한 상태다.

```text
pytest
-> 18 passed
```

하지만 현재 `Backend_Server` 코드 기준으로는 아직 RAG HTTP 호출이 직접 연결돼 있지 않다.
즉 RAG는 준비되어 있지만, 백엔드 오케스트레이션은 별도 작업이 필요하다.

## 5. Backend 담당자 기준 핵심 포인트

- Backend DB가 source of truth다.
- RAG는 검색용 파생 데이터를 저장한다.
- Backend는 `중간 산출물(extracted_text, canonical_text, chunk, embedding)`을 받을 필요가 없다.
- MQ로 전환할 경우 Backend는 `RAG가 보내는 진행 상태 이벤트`를 받아 job 상태를 갱신해야 한다.
- Backend는 최종적으로 아래 결과만 받으면 된다.

```text
/knowledge/ingest
- status
- indexed_document_count
- indexed_chunk_count
- indexed_source_ids

/templates/index
- indexed_template_count
- template_ids

/templates/match
- template candidate 목록

q.2rag.draft
- templates[]
```

### 5-1. MQ로 전환할 때 상태를 아는 방법

Backend가 각 단계를 직접 계산하는 게 아니라,
RAG가 보내는 progress / completed / failed 이벤트를 받아서 안다.

```text
Frontend
  -> Backend에 작업 시작 요청

Backend
  -> job_id 생성
  -> DB에 jobs row 생성
  -> MQ에 요청 발행

RAG
  -> 처리 단계 변경
  -> progress 이벤트 발행

Backend
  -> progress 이벤트 consume
  -> jobs 테이블 업데이트
  -> SSE 또는 polling으로 사용자에게 전달
```

즉 역할 분리는 아래와 같다.

```text
RAG
- 실제 처리 수행
- 진행 상태 이벤트 발행

Backend
- job 상태 저장
- 사용자에게 상태 전달
```

권장 최소 Job 테이블 구조:

```text
jobs
- job_id
- request_id
- user_id
- job_type
- target_type
- target_id
- status
- progress_step
- progress_message
- error_code
- error_message
- payload_json
- created_at
- updated_at
- completed_at
```

즉 MQ를 쓰더라도, 사용자에게 보이는 진행 상태의 source of truth는 RabbitMQ가 아니라 Backend DB다.

권장 상태 예시:

```text
knowledge/ingest
- QUEUED
- DOWNLOADING
- EXTRACTING
- CHUNKING
- EMBEDDING
- INDEXING
- COMPLETED
- FAILED

templates/match
- QUEUED
- CANONICALIZING
- EMBEDDING
- MATCHING
- COMPLETED
- FAILED

draft
- QUEUED
- RETRIEVING_CONTEXT
- GENERATING
- EMBEDDING_REPLY
- COMPLETED
- FAILED
```

### 5-2. knowledge ingest 입력 매핑 메모

현재 `rag /knowledge/ingest`는 외부 계약에서 manual을 `PDF 전용`으로 본다.
즉 Backend는 manual 요청에 `media_type`이나 `file_type`을 보낼 필요가 없다.

정리하면:

```text
외부 요청
- source_id
- file_name
- local_path 또는 presigned_url
- title
- metadata

RAG 내부
- media_type = application/pdf 로 고정 정규화
```

즉 이 구간은 백엔드와 타입 필드를 맞추는 문제보다,
`업로드 문서가 현재 PDF only`라는 계약을 명확히 유지하는 것이 더 중요하다.

## 6. AI 담당자 기준 핵심 포인트

현재 RAG가 가장 기대하는 AI 입력은 아래 정도다.

```text
- summary
- intent
- domain
- entities (optional)
- keyphrases (optional)
```

이 값들은 `templates/match`와 `draft` 품질을 높이기 위한 입력이다.
다만 현재 `rag/`는 raw email 필드만 있어도 내부 canonicalization으로 최소 동작은 가능하다.

## 7. 인프라 담당자 기준 핵심 포인트

현재 구현은 `HTTP API 서버 + Chroma local persistent storage` 기준이다.

```text
RAG 서버
  -> FastAPI
  -> Chroma PersistentClient
  -> .rag-data/chroma
```

자원 특성:

- GPU 필수 아님
- CPU / 메모리 / 디스크 중심
- 이미지형 PDF 대응을 위한 OCR을 붙이면 CPU 사용량이 더 커질 수 있음

## 8. RabbitMQ 전환 관점

현재 구현은 HTTP만 있다.
다만 아래 작업은 MQ 전환 후보가 될 수 있다.

```text
MQ로 전환하기 좋은 것
- knowledge/ingest
- templates/index
- templates/match
- draft

HTTP 유지가 더 자연스러운 것
- health
- 사용자가 직접 누르는 재추천 / 재생성 시작 요청
```

이유:

- `knowledge/ingest`, `templates/index`, `draft`는 시간이 걸리거나 재시도가 중요하다.
- `templates/match`도 메일 수신 파이프라인에서 자동 수행한다면 MQ가 더 자연스럽다.
- 즉 자동 파이프라인은 MQ, 직접 조회/제어는 HTTP로 나누는 것이 현재 서비스 흐름과 더 잘 맞는다.

## 9. 현재 저장 구조 요약

RAG는 별도 비즈니스 테이블을 만들지 않고, Chroma 내부 구조를 사용한다.

```text
.rag-data/chroma
  ├─ chroma.sqlite3
  ├─ <vector segment uuid>/
  └─ ...
```

관리 기준은 파일명이 아니라 아래다.

```text
collection
-> namespace
-> record_id
-> metadata
-> payload_json
```

예:

```text
knowledge record
- faq:faq-1:0
- manual:manual-1:0

template record
- template:101
```

상세 구조는 [Chroma-저장구조-ERD.md](./Chroma-저장구조-ERD.md)를 본다.
